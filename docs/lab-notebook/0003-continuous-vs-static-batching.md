# 0003 · 实验1:连续批处理 vs 静态批处理(inflight ↔ V1)★最关键

> 实验台用法见 [`../L2-LAB.md`](../L2-LAB.md)。这是消融矩阵第一行,也是 M2 的核心。

| 字段 | 内容 |
|---|---|
| 日期 | 2026-06-21 |
| 里程碑 | M2(调度器 / 连续批处理) |
| 旋钮 | `gpt_model_type`:`inflight_fused_batching` → `V1`(R) |
| 引擎/层 | small(0.5B),直连 engine(绕开 BLS/护栏) |

## ① 假设
关掉连续批处理、换成静态批(V1):吞吐会暴跌、延迟会飙升。因为静态批要等**一整批里最长的那条**做完才放新请求进来(head-of-line blocking,"包车 vs 拼车")。

## ② 预测(动手前)
吞吐掉一半以上;TTFT(首 token)明显变高,尤其高并发时。

## ③ 实验设置
`perf_benchmark.py` 直连 small 引擎,C=1/8/32,**变长输出 16–256**(变长才暴露 head-of-line)。
基线 = `inflight_fused_batching` + `max_utilization`;消融 = `V1`。
> **动手才踩到的坑(预测里没有):** V1 在 0.14 已被标记 **deprecated**,而且**只能配 `guaranteed_no_evict`**,否则直接拒绝加载(`V1 only supports GUARANTEED_NO_EVICT scheduling policy`)。所以消融实际是 `V1 + no_evict`。本工作负载 KV 没压满,`no_evict` 与 `max_util` 行为基本一致,所以差异几乎全来自批处理方式本身。

## ④ 实测
| 并发 | 吞吐 inflight | 吞吐 V1 | 首token inflight | 首token V1 |
|---|---|---|---|---|
| C=1  | 197 tok/s  | 191 tok/s  | 7 ms   | **603 ms** |
| C=8  | 1239 tok/s | **545 tok/s** | 10 ms  | **2064 ms** |
| C=32 | 4099 tok/s | **1965 tok/s** | 11 ms  | **2112 ms** |

**吞吐(C=32):连续批 ≈ 2.1 倍**
```
inflight  ████████████████████████████████████████  4099 tok/s
V1 静态批 ███████████████████                        1965 tok/s   ← 砍半
```

**首 token 延迟(C=32):连续批 ≈ 195 倍快**
```
inflight  ▏ 11 ms
V1 静态批 ████████████████████████████████████████ 2112 ms   ← 用户要等 2 秒才看到第一个字
```

**引擎同时在跑的请求数(C=32 采样)——这张图就是 head-of-line:**
```
inflight  ≈ 一直 16–32 满载         （谁下车,座位立刻给新人）
V1        31 ███...  → 1 █           （31 个一起上车,30 个早早做完却空转,
            └ 在 31 和 1 之间锯齿跳    只剩最长那条在跑,新请求被挡在门外)
          采样:46 次在 31,76 次在 1  ← 大量时间只有 1 条在跑、GPU 闲着
```
> 还有个意外:V1 的 **ITL = NaN** —— 它把整段输出**一次性**吐回,根本不增量流式。所以上面 V1 的"首 token"其实 = **整段生成完**的时间。对用户体验是双杀:既慢一半,又得干等到全部生成完才看到任何字。

## ⑤ Gap 分析
方向全对(吞吐砍半、延迟飙升),但有两个预测里没有的收获:
1. **V1 强制 `no_evict`** —— 静态批不做驱逐,这个耦合是动手才知道的。
2. **V1 不增量流式** —— 整段一次返回。这比"吞吐砍半"对用户更致命。
锯齿图(31↔1)把 head-of-line blocking 直接画了出来:一批 31 个一起进,短的早早做完但槽位不能复用,只剩最长那条在跑、其余 30 个槽空转,直到整批结束才放新请求。

## ⑥ vLLM 机制
对照 `vllm/v1/core/sched/scheduler.py` 的 `schedule()`:它是 **iteration-level**——每一步都重新决定 admit/evict,谁结束了立刻补新请求进来。这正是 inflight 稳定满载、而 V1 锯齿空转的根因。

## ⑦ 结论(大白话)
- **连续批处理 = 拼车**:谁到站下车,座位立刻让给等待的新人 → GPU 一直坐满 → **吞吐 2 倍、首字 10ms、边生成边吐字**。
- **静态批处理 = 包车**:一车人必须等最后一个到站才发下一班 → 30 个空座干等 → **吞吐砍半、首字等 2 秒、还不能边走边吐字**。
- 这就是为什么 vLLM/TRT-LLM 默认连续批处理,NVIDIA 干脆把 V1 标记 deprecated。**"★最关键"名副其实。**
