# 0002 · 实验台就位 + 「测对层」(直连引擎 vs 穿过网关)

> 实验台用法见 [`../L2-LAB.md`](../L2-LAB.md)。这条把第 6 节的「测量脊梁」补上(`client/perf_benchmark.py`),并验证第 4.1 节「测对层」这条铁律。

## ① 假设
要量「引擎本身」的并发能力,**必须直连引擎、绕开 BLS+护栏**。穿过 BLS 测到的是「网关+护栏+引擎」混在一起——而护栏是 `count:1` 的单实例,会把请求**串行**化,把引擎的并发能力压下去。

## ② 预测(动手前)
- 直连引擎、16 并发 → 引擎应能同时处理接近 16 条(峰值活跃请求 ≈ 16)。
- 穿过 BLS、16 并发 → 会被单实例护栏卡住,引擎侧峰值活跃远小于 16(上一轮实测 ≈ 3)。

## ③ 实验设置
新写的 `client/perf_benchmark.py`(闭环压测:C 路并发跑完 N 个请求,量 TTFT/ITL/吞吐/P50/P99)。直连引擎用**裸 token id**,固定输出长度(不传 end_id → 长度确定,吞吐干净)。同时每 0.05s 采样引擎 `nv_trt_llm_request_metrics{request_type="active"}`。

```bash
.venv/bin/python client/perf_benchmark.py --target engine --model tensorrt_llm_small \
    --concurrency 16 --num-requests 64 --input-len 128 --output-len 128
```

## ④ 实测
| 路径 | 并发 | 峰值活跃请求 | 吞吐 | TTFT p50 |
|---|---|---|---|---|
| **直连引擎** | 16 | **16**(47/56 采样都在 16) | 2631 tok/s | 25 ms |
| 穿过 BLS(上一轮) | 16 | **3** | — | — |

**「测对层」一图胜千言 —— 同样 16 并发,峰值同时在跑的请求数:**
```
直连引擎    ████████████████  16   ← 引擎本身轻松吃满
穿过 BLS    ███                3   ← 卡在 count:1 的单实例护栏(串行)
```

**连续批处理把吞吐随并发拉起来(直连引擎,输出 16–256 变长):**
```
吞吐 (tok/s) vs 并发
C=1   ██                                        197
C=8   █████████████                            1239
C=32  ████████████████████████████████████████ 4099     (≈ C=1 的 21 倍)
```

## ⑤ Gap 分析
预测完全对上:直连引擎峰值活跃 = 16(吃满我给的并发),穿过 BLS 只有 3。**差距 = 单实例护栏的串行**。这也正是上一轮把 BLS 改成线程池后、引擎侧并发仍只有 3 的原因——瓶颈已经从 BLS 移到了护栏(`AI-INFRA-DIRECTION` §1.1 早就点名的「护栏单实例串行」)。

## ⑥ 结论(大白话)
1. **测引擎要直连引擎**,别穿过网关——否则你量到的是护栏的串行,不是引擎的本事。
2. **连续批处理是真的有用**:并发 1→32,吞吐 197→4099 tok/s,涨了 21 倍(单条请求几乎没变慢,TTFT 6.7→10.8ms)。这就是「拼车」的威力。
3. 端到端要更快,下一个要扩的是**护栏实例数**(把 `count:1` 调大),它现在是 BLS 路径的串行瓶颈。
