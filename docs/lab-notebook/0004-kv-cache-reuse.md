# 0004 · 实验2:KV 缓存复用(共享前缀)

> 实验台用法见 [`../L2-LAB.md`](../L2-LAB.md)。消融矩阵第二行(M3)。

| 字段 | 内容 |
|---|---|
| 日期 | 2026-06-21 |
| 里程碑 | M3(KV 缓存 / PagedAttention) |
| 旋钮 | `enable_kv_cache_reuse`:`true` → `false`(R) |
| 引擎/层 | small(0.5B),直连 engine |

## ① 假设
当多个请求**共享同一段开头**(比如同一段长 system prompt)时,开启 KV 复用能让第 2 个之后的请求**跳过重复的 prefill**,首 token(TTFT)大幅变快;关掉则每个请求都得把整段开头重算一遍。

## ② 预测(动手前)
共享前缀越长,reuse ON 的 TTFT 越低;关掉后 TTFT ≈ 整段 prefill 时间。

## ③ 实验设置
直连 small 引擎。**专门设计的负载**:输入 4000 token,其中**前 3968 token 是所有请求完全相同的共享前缀**,只有结尾 32 token 不同;输出只要 8 token(把焦点放在 prefill/TTFT,不看 decode)。C=1 与 C=4。
> 关键:KV 复用**只有共享前缀才看得到**——随机 prompt 完全没有效果(L2-LAB §4.2「负载要为旋钮量身定制」)。

## ④ 实测
| 配置 | 并发 | TTFT p50 | TTFT p99 | req/s | reused 块 |
|---|---|---|---|---|---|
| **reuse ON** | C=1 | **8.5 ms** | 264.6 ms | 11.9 | 2418 |
| reuse OFF | C=1 | 81.8 ms | 257.6 ms | 6.7 | 0 |
| **reuse ON** | C=4 | **14.6 ms** | 33.2 ms | **61.9** | 2418 |
| reuse OFF | C=4 | 264.1 ms | 350.6 ms | 10.0 | 0 |

**TTFT p50(越低越好)—— 复用直接把"重算开头"这步省掉:**
```
C=1   ON  ██                 8.5 ms
      OFF ████████████████   81.8 ms      (~10×)

C=4   ON  █                  14.6 ms
      OFF ████████████████████████████████████████  264.1 ms   (~18×)
```
- `reused` KV 块:ON=**2418**,OFF=**0** —— 直接证明复用真的发生了。
- C=4 时差距更大(18×):关掉复用后,4 个 4000-token 的 prefill **同时抢算力**,每个都更慢;开着复用则只有头一个冷算、其余直接命中缓存。

## ⑤ Gap 分析
方向完全对上。一个细节值得记:
- reuse **ON、C=1**:p50=8.5ms(命中缓存,跳过 prefill),但 **p99=265ms** —— 那是**第一个冷请求**,缓存还没建立,得老老实实算完整 4000-token prefill。"第一个人铺路,后面的人白嫖。"
- reuse **OFF、C=1**:p50=81.8ms 而非 265ms —— 因为**kernel/上下文预热**(第一个请求 265ms 是冷 kernel,之后 ~82ms 是热 kernel 但仍要算完整 prefill)。**注意区分两种"冷":kernel 预热 ≠ KV 复用。** 真正的复用收益 = 8.5ms vs 82ms ≈ 10×。

## ⑥ vLLM 机制
对照 `vllm/v1/core/kv_cache_manager.py` + block hashing:把前缀按块做哈希,相同哈希的块直接复用(引用计数 +1),省掉那部分 prefill。这正是 `reused=2418` 的来源。

## ⑦ 结论(大白话)
- **同一段长开头被很多请求共用时(典型:固定 system prompt / few-shot 例子 / RAG 里同一段文档),KV 复用 = 第一个人算好开头,后面的人直接抄** → 首字快 10–18 倍,单位时间能服务的请求多 ~6 倍。
- 但**只对共享前缀有效**;请求各不相同就没用。所以这是「把固定前缀放最前面、让大家共享」的工程设计动机(本 repo 的 chat 模板正是把共享部分放最前)。
