# 0008 · 实验6:max_batch_size(并发天花板)— 第一个 build 级实验

> 实验台用法见 [`../L2-LAB.md`](../L2-LAB.md)。消融矩阵第六行(B,需**重编引擎**)。直接验证 [M0](./0001-m0-kv-memory.md) 的「第二个天花板 = max_batch_size」。

| 字段 | 内容 |
|---|---|
| 日期 | 2026-06-21 |
| 里程碑 | M2 / M0 |
| 旋钮 | `max_batch_size`(build 期):16 / 64(基线) / 128 |
| 引擎/层 | small(0.5B),直连 engine |

## ① 假设
M0 发现两个并发天花板:KV 容量 与 `max_batch_size`。短序列时是 **max_batch_size 先到顶**。把它调小/调大,并发上限就跟着平移。

## ② 预测(动手前)
- 并发 ≤ max_batch_size:吞吐随并发涨,TTFT 平。
- 并发 > max_batch_size:多出来的请求**排队** → 吞吐封顶、TTFT 飙。
- 调大 max_batch_size 把天花板抬高 → 吞吐继续涨(直到 GPU 算力饱和)。

## ③ 实验设置
**重编引擎**(reuse 现成的 `ckpt/0.5b`,只跑 `trtllm-build --max_batch_size {16,128}`,各 8 秒)。把 `gpt_model_path` 指过去 + Triton `max_batch_size` 对齐 + 重启。固定负载:in128 + 变长 out 16–256。日志确认 `maxNumSequences` = 16/64/128。

## ④ 实测
**吞吐(tok/s)vs 并发,三个引擎:**
| 并发 C | bs16 | bs64(基线) | bs128 |
|---|---|---|---|
| 16  | 2202 | — | — |
| 32  | 2516 | 3596 | 3810 |
| 64  | ~2533(封顶) | 6086 | 5999 |
| 128 | ~2533(封顶) | 6630 | 6999 |

```
吞吐随并发(每条在自己的 max_batch_size 处封顶):
bs16   ██▓▓▓▓ 2202→2516→2533→2533   封顶@16,之后全在排队
bs64   ░░███████████ 3596→6086→6630  封顶@64
bs128  ░░████████████ 3810→5999→6999 封顶@128
```

**TTFT p50 在"超过天花板"时爆炸(= 排队):**
```
bs16  @C=32 (>16) :  771 ms     ← 16 跑、16 排队
bs16  @C=48 (>16) : 1612 ms
bs64  @C=128(>64) : 1097 ms     ← 64 跑、64 排队
bs128 @C=128(=128):   55 ms     ← 128 全跑,不排队
```

## ⑤ Gap 分析
预测全中,还白送一个洞见:
- **抬高天花板对"吞吐"的收益会衰减**:bs64→bs128 在 C=128 只多 **+5.6%** 吞吐(6630→6999)——因为 GPU 算力已接近饱和(C=64→128 增幅从 +69% 掉到 +17%)。
- **但对"延迟"收益巨大**:同样 C=128,bs64 的 TTFT 1097ms vs bs128 的 55ms = **20× 改善**——因为 bs128 不用排队。
- 所以"调大 max_batch_size"在高并发下**主要是省延迟(免排队),其次才是吞吐**(吞吐受 GPU 算力封顶)。

## ⑥ 结论(大白话)
- `max_batch_size` = **能同时上车的人数上限**。坐满之前,加人=加吞吐、不排队;一旦坐满,多出来的人**站台排队**→ 吞吐不再涨、首字时间爆炸。
- 调大它 = 加座位:高并发下**先救延迟**(免排队),吞吐则到 GPU 算力上限就封顶。
- 这是 build 期参数(改了要重编引擎,8 秒),不像前 5 个 runtime 旋钮改配置即可。**M0 的「两个天花板」现在两条都实测过了**(KV 容量见 [0007](./0007-kv-mem-fraction.md),batch_size 见这条)。
