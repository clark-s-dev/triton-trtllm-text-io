# 0005 · 实验3:分块上下文(长 prefill 会不会卡住别人吐字)

> 实验台用法见 [`../L2-LAB.md`](../L2-LAB.md)。消融矩阵第三行(M3)。**这条的结论是"温和 + 有噪声",顺便上了一课关于"怎么测才不被骗"。**

| 字段 | 内容 |
|---|---|
| 日期 | 2026-06-21 |
| 里程碑 | M3 |
| 旋钮 | `enable_chunked_context`:`true` → `false`(R) |
| 引擎/层 | small(0.5B),直连 engine |

## ① 假设
关掉分块上下文:一条超长输入的 prefill 会**一口气占满一个迭代**,把正在吐字的其他流卡住 → 它们的 ITL(字间延迟)出现尖刺。开着则把长 prefill 切块、与 decode 交织 → 别人吐字平滑。

## ② 预测(动手前)
chunked OFF 的 ITL p99(最坏字间延迟)会**明显高于** ON。

## ③ 实验设置
**专门设计的"碰撞"负载**:输入 128–4000 变长(长的就是 prefill 炸弹)+ 输出 256(让请求持续 decode),C=16。
> **踩坑 → 改方法:** 第一次只各跑 1 次,得到 ON=114ms、OFF=90ms,看起来"OFF 反而更好"——但 ITL p99 噪声极大。于是改成**每个配置跑 5–6 次取分布**。

## ④ 实测(ITL p99,每配置 5–6 次)
| 配置 | ITL p99 各次 (ms) | 中位 | 吞吐 | TTFT p99 中位 |
|---|---|---|---|---|
| **chunked ON**  | 22, 28, 54, 64, 69 | **~54** | ~1530 tok/s | ~716 ms |
| chunked OFF | 18, 63, 87, 88, 90, 106 | **~88** | ~1560 tok/s | ~550 ms |

```
最坏字间延迟 ITL p99(越低=吐字越平滑):
chunked ON   [22 ──●54── 69] ms          中位 54
chunked OFF  [18 ─────●88───── 106] ms    中位 88   (ON 中位低 ~1.6×,但区间重叠)

吞吐:两者基本一样(~1530 vs ~1560 tok/s)
TTFT p99:OFF 略低(~550 vs ~716)← 注意这是个权衡(见下)
```

## ⑤ Gap 分析(本条最值钱的部分)
1. **方向对了,但远没预测的那么夸张。** ON 确实让最坏 ITL 低了 ~1.6×(中位 54 vs 88),不是连续批/KV 复用那种 2×/10× 的悬殊。
2. **差点被单次测量骗了。** 单跑一次是 ON=114、OFF=90,会得出"分块没用甚至有害"的**错误结论**;跑 5–6 次看分布才看出 ON 其实更平滑。**教训:噪声大的尾延迟指标(p99)必须多跑取分布,别拿单次对比。**
3. **为什么效果温和?**(§4.4 小模型陷阱)0.5B 上一个 4000-token 的 prefill 只要 ~80ms,不够长到把 decode 饿死;而且 inflight batcher 本来就会把 prefill 和 decode 凑一批跑。**分块上下文真正的舞台是大模型 + 超长上下文(32K+)**:那时一次性 prefill 会卡好几秒,分块才救命。
4. **看到一个权衡:** chunked OFF 的 TTFT p99 略低(prefill 一口气算完,那条请求自己的首字反而快)——代价是把别人的吐字卡了。**分块 = 牺牲"自己首字早一点"换"大家吐字都平滑"。**

## ⑥ vLLM 机制
对照 vLLM 的 chunked-prefill(`long_prefill_token_threshold`):超过阈值的 prefill 被切成块,和 decode 在调度里交织。机制和 TRT-LLM 一致。

## ⑦ 结论(大白话)
- 分块上下文**确实**让"长输入不卡别人吐字",但在这台小模型上只是温和效果(最坏 ITL ~1.6×),还被噪声盖住一半。
- 它是为**大模型 / 超长上下文**准备的功能;小模型短上下文下,开不开关系不大。
- **方法学收获 > 数值收获**:p99 这种尾指标,**单次会骗人,要取分布**。
