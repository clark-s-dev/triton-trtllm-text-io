# 0007 · 实验5:KV 显存占比(池子多大 → 能并发多少)

> 实验台用法见 [`../L2-LAB.md`](../L2-LAB.md)。消融矩阵第五行(M3),也直接验证了 [M0](./0001-m0-kv-memory.md) 的「KV 容量天花板」。

| 字段 | 内容 |
|---|---|
| 日期 | 2026-06-21 |
| 里程碑 | M3 |
| 旋钮 | `kv_cache_free_gpu_mem_fraction`:扫 0.25 → 0.1 → 0.05(R) |
| 引擎/层 | small(0.5B),直连 engine,固定负载 C=64 in2000+out500 |

## ① 假设
这个值直接决定 KV 池有多大 → 能同时塞下多少请求。太低 → 请求排队(吞吐悬崖);太高 → 抢别的引擎显存 / OOM。

## ② 预测(动手前)
池子 ≥ 负载需求时,吞吐/延迟平;一旦池子 < 需求,排队 → 吞吐掉、TTFT 飙。

## ③ 实验设置
固定负载(C=64,in2000+out500 ≈ 需要 2496 个 KV 块),只扫 `kv_cache_free_gpu_mem_fraction`。每次读启动后的 `kv_cache_block_metrics{type="max"}` 拿到真实池子大小。

## ④ 实测
| fraction | 池子(块) | 池子(token) | 吞吐 | TTFT p50 | peak paused |
|---|---|---|---|---|---|
| 0.25 | 6857 | ~439K | 3057 tok/s | 365 ms | 0 |
| 0.10 | 2743 | ~175K | 3029 tok/s | 367 ms | 0 |
| 0.05 | 1372 | ~88K | **2346 tok/s** | **2004 ms** | 1 |

**池子大小 = fraction × 常数(完美线性,直接呼应 M0):**
```
6857 : 2743 : 1372  ==  0.25 : 0.10 : 0.05     ✓  (比例完全一致)
```
**吞吐 vs fraction —— 平台,然后悬崖:**
```
0.25  ████████████████████████████████  3057   池 439K,负载装得下
0.10  ███████████████████████████████   3029   池 175K,刚好装下
0.05  █████████████████████████         2346   池 88K,装不下 → 排队   ← 悬崖 (−23%)
```
**TTFT p50 —— 装得下时平,装不下时飙:**
```
0.25  ▏365 ms
0.10  ▏367 ms
0.05  ████████████████████ 2004 ms     ← 池子不够,请求排队等 KV (5.5×)
```

## ⑤ Gap 分析
预测完全对上,而且白送两个洞见:
1. **池子大小严格线性正比于 fraction**(6857/2743/1372 与 0.25/0.1/0.05 比例一致)—— 这正是 M0 手算 KV 的延续:`池子 token = fraction × 可用显存 / (字节/token)`。
2. **悬崖位置可预测**:负载要 2496 块,所以 fraction < 2496/6857×0.25 ≈ **0.09** 就开始排队。0.1(池 2743)刚好在悬崖上沿(还平),0.05(池 1372)已跌下去。
> 没测「太高」那端(fraction→0.9):那是**另一个失败模式**——会和 large 引擎抢显存、或 OOM(M0 已算过两引擎共 ~17.9/22.5GB,没多少余量)。

## ⑥ vLLM 机制
对照 vLLM:`num_gpu_blocks = (可用显存 × gpu_memory_utilization) / 每块字节`,再 `max_num_seqs` 封顶。fraction 越大 → blocks 越多 → 能并发的序列越多。

## ⑦ 结论(大白话)
- `kv_cache_free_gpu_mem_fraction` 就是「给 KV 缓存留多大的池子」,**池子大小和这个值成正比**。
- 池子 ≥ 负载需求 → 一切照常;**池子 < 需求 → 请求开始排队,吞吐掉、首字飙(本次 −23% / 5.5×)**。
- 调参取舍:**太低 = 排队悬崖,太高 = 抢显存/OOM**;甜点是「刚好装下你的峰值并发」。
- 这条把 M0 的「KV 容量天花板」从手算变成了实测曲线。
